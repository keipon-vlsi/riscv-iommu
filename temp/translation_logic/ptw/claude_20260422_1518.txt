  ---
  各信号が1になる条件

  is_store_i
  - trans_type_i[1:0] == 2'b11 かつ trans_type_i[3] == 0 のとき
  - 対応するトランザクション: UNTRANSLATED_W または TRANSLATED_W
  - 意味：Write / AMO（Atomic Memory Operation）アクセス のとき1になる
  - トップレベル (riscv_iommu.sv:906) では AXI の書き込みチャネル (AW)
  のリクエストが来たとき trans_type = UNTRANSLATED_W がセットされる

  is_rx_i
  - trans_type_i[0] == 1 かつ trans_type_i[1] == 0 かつ trans_type_i[3] == 0
  のとき
  - 対応するトランザクション: UNTRANSLATED_RX または TRANSLATED_RX
  - 意味：Read-for-Execute（命令フェッチ） アクセスのとき1になる
  - トップレベル (riscv_iommu.sv:866) では AXI 読み込みチャネル (AR) の
  AxPROT[2] == 1（= Instruction access）のとき trans_type = UNTRANSLATED_RX
  がセットされる

    pscid   [19:0]       ← プロセス空間 ID
    gscid   [15:0]       ← 仮想マシン ID
    vpn2    [GPPN2:0]    ← 11-bit (Sv39x4 拡張)
    vpn1    [8:0]
    vpn0    [8:0]
    is_1S_2M / is_1S_1G  ← S1 スーパーページ種別
    is_2S_2M / is_2S_1G  ← S2 スーパーページ種別
    is_msi               ← MSI エントリ
    en_1S / en_2S        ← 登録時のステージ有効状態
    valid

  content_q[i]:
    pte_1S  (riscv::pte_t)  ← S1 リーフ PTE（ppn = GPA の PPN）
    pte_2S  (riscv::pte_t)  ← S2 リーフ PTE（ppn = SPA の PPN）

  S1・S2 の PTE
  が1エントリに同居するため、2ステージ翻訳を1回のルックアップで完結できる。

  ---
  3. PTE 登録の仕方

  条件: update_i && replace_en[i] && (S1有効→S1_PTE.v || S2有効→S2_PTE.v)

  登録対象スロット i は後述の PLRU
  で決定される。タグとコンテントを同時に書き込む。

  tags_n[i] = '{
      pscid:    up_pscid_i,
      gscid:    up_gscid_i,
      vpn2/1/0: up_vpn_i の各フィールド,
      en_1S:    en_1S_i,  en_2S:    en_2S_i,
      is_1S_1G: up_1S_1G_i, ...
      valid:    1'b1
  };
  content_n[i].pte_1S = up_1S_content_i;
  content_n[i].pte_2S = up_2S_content_i;

  ---
  4. ヒット・ミス判定ロジック

  全エントリを並列に比較する（組み合わせ回路）。

  ヒット条件（全て AND）:
    ① valid == 1
    ② PSCID 一致   : (lu_pscid == pscid || pte_1S.g) && en_1S_i  ||  !en_1S_i
    ③ GSCID 一致   : (lu_gscid == gscid && en_2S_i)              ||  !en_2S_i
    ④ ステージ一致 : en_1S_i == tag.en_1S && en_2S_i == tag.en_2S
    ⑤ VPN2 一致    : vpn2 == tag.vpn2 & mask_pn2
                     ※ mask_pn2: 1S有効→9-bit (Sv39), 2Sのみ→11-bit (Sv39x4)
    ⑥ ページサイズ別 VPN 比較:
         1G: ① VPN2 のみ一致 (VPN1/VPN0 は offset 扱い)
         2M: ① VPN2 + VPN1 一致 (VPN0 は offset 扱い)
         4K: ① VPN2 + VPN1 + VPN0 全一致

  is_trans_1G / is_trans_2M
  はステージの組み合わせを考慮してページサイズを判定する：

  is_trans_1G = ((is_1S_1G && S1_en) || !S1_en) && ((is_2S_1G && S2_en) ||
  !S2_en)
    → 有効な全ステージが 1G の場合のみ 1G 扱い

  is_trans_2M = S1+S2 両有効時:
                 (is_1S_2M && (is_2S_1G||is_2S_2M)) || (is_2S_2M &&
  (is_1S_1G||is_1S_2M))
               それ以外: (is_1S_2M && S1_en) || (is_2S_2M && S2_en)

  ---
  5. 権限チェックの判定ロジック

  IOTLB 内には権限チェックなし。ヒット時に PTE
  をそのまま出力し、チェックは上位の wrapper (rv_iommu_tw_sv39x4_pc.sv) で行う。

  IOTLB ヒット後の wrapper での判定:

  【S1 権限チェック (lu_1S_content_o に対して)】
    ストア && !pte_1S.w  → STORE_PAGE_FAULT
    is_rx  && !pte_1S.x  → LOAD_PAGE_FAULT
    U-mode && !pte_1S.u  → LOAD/STORE_PAGE_FAULT
    S-mode && pte_1S.u && (!SUM || pte_1S.x) → LOAD/STORE_PAGE_FAULT

  【S2 権限チェック (lu_2S_content_o に対して)】
    ストア && !pte_2S.w  → STORE_GUEST_PAGE_FAULT
    is_rx  && !pte_2S.x  → LOAD_GUEST_PAGE_FAULT

  ---
  6. 無効化（Flush）ロジック

  IOTINVAL.VMA（S1 エントリ対象）

  {flush_gv_i, flush_av_i, flush_pscv_i} の 8 パターンで対象を絞り込む。

  ┌────────────┬────────────────────────────────────────┐
  │ GV|AV|PSCV │               無効化対象               │
  ├────────────┼────────────────────────────────────────┤
  │ 000        │ 全ホスト空間エントリ（2S無効・1S有効） │
  ├────────────┼────────────────────────────────────────┤
  │ 001        │ PSCID 一致（G=1 除外）                 │
  ├────────────┼────────────────────────────────────────┤
  │ 010        │ IOVA 一致（G=1 含む）                  │
  ├────────────┼────────────────────────────────────────┤
  │ 011        │ IOVA + PSCID 一致（G=1 除外）          │
  ├────────────┼────────────────────────────────────────┤
  │ 100        │ GSCID 一致（2S+1S 両有効）             │
  ├────────────┼────────────────────────────────────────┤
  │ 101        │ GSCID + PSCID 一致（G=1 除外）         │
  ├────────────┼────────────────────────────────────────┤
  │ 110        │ GSCID + IOVA(GVA) 一致                 │
  ├────────────┼────────────────────────────────────────┤
  │ 111        │ GSCID + PSCID + IOVA 一致（G=1 除外）  │
  └────────────┴────────────────────────────────────────┘

  IOTINVAL.GVMA（S2 エントリ対象）

  {flush_gv_i, flush_av_i} の 3 パターン。

  ┌───────┬─────────────────────────┐
  │ GV|AV │       無効化対象        │
  ├───────┼─────────────────────────┤
  │ 0x    │ 全 S2 有効エントリ      │
  ├───────┼─────────────────────────┤
  │ 10    │ GSCID 一致              │
  ├───────┼─────────────────────────┤
  │ 11    │ GSCID + GPA（ADDR）一致 │
  └───────┴─────────────────────────┘

  GVMA の GPA 一致判定は、タグに GPA を直接持たないため make_gppn() で
  pte_1S.ppn とタグの VPN からエントリの GPA
  を再構築して比較する（ハードウェアコストあり、仕様通りの実装）。

  ---
  7. 置換アルゴリズム（PLRU）

  Pseudo Least Recently Used（疑似 LRU） をバイナリツリーで実装。

  エントリ数 N に対して 2*(N-1) ビットのツリーノードを持つ。

           0          ← root
          / \
         1   2
        / \ / \
       3  4 5  6      ← (N=8 の例)

  - ヒット時：ヒットしたエントリへのパスを「最近使用」方向にビット更新
  - 置換選択時：ツリーを root
  から葉に向かってたどり、「最も使われていない」エントリを replace_en でアサート
